# ROADMAP

문서 상태: Draft v0.1  
문서 목적: 단계별 개발 순서, 마일스톤, 산출물, 진입/종료 조건 정의

## 1. 로드맵 전제

이 로드맵은 다음을 전제로 한다.

- 소규모 인디 팀
- 싱글 플레이 우선
- 시뮬레이션 재미 검증이 최우선
- 그래픽 완성도보다 구조 안정성과 관찰 재미를 먼저 확보
- 자동 앱 감지, 멀티플레이, 복잡한 경제 시스템은 후순위

## 2. 핵심 원칙

1. **작게 시작하되 버리지 않는 구조로 만든다.**
2. **보이는 재미를 먼저 만든다.**
3. **콘텐츠보다 시스템의 관찰 밀도를 먼저 검증한다.**
4. **AI는 마지막에 붙여도 되게 한다.**
5. **멀티플레이는 구조만 열어 두고 실제 구현은 미룬다.**

## 3. 개발 단계 요약

| 단계 | 기간(권장) | 목표 | 핵심 결과물 |
| --- | --- | --- | --- |
| M0 | 1~2주 | 프리프로덕션 정렬 | 문서, 레포 구조, 기준 기술 검증 |
| M1 | 2~3주 | 공간/렌더링 수직 슬라이스 | 타일맵, 카메라, A 이동 |
| M2 | 3~4주 | 자율 시뮬레이션 코어 | 틱, 욕구, 루틴, 행동 선택 |
| M3 | 3~4주 | 사용자 입력 반영 | 공부/게임/운동/휴식 입력이 장면 변화로 연결 |
| M4 | 3~4주 | 관계/이벤트/해금 | A↔B, 사회적 장면, 해금 1종 |
| M5 | 3~5주 | 저장/안정화/콘텐츠 확장 | 저장 슬롯, B/C, 확장 시설 |
| M6 | 2~4주 | AI 보조 연출 및 제품 폴리시 | 일기/요약/짧은 대사 |
| M7 | 이후 | 클라우드/비동기 공유/멀티 시드 | 후속 확장 검토 |

아래 세부 내용은 각 마일스톤을 “실제로 출시 가능한 중간 결과물” 기준으로 적는다.

## 4. 마일스톤 상세

### M0. 프리프로덕션 정렬

**목표**  
문서와 구조를 먼저 고정해, 이후 구현 시 방향 드리프트를 막는다.

**포함 범위**

- PRD, 아키텍처, 데이터 모델, 게임 루프 합의
- 레포 구조와 패키지 경계 정의
- 에셋 파이프라인/Tiled 맵 작업 방식 정의
- 기본 저장 포맷 초안 정의
- 팀 내부 용어 통일

**종료 기준**

- 문서 세트가 레포에 반영됨
- 주요 ADR 4개 초안 작성
- Phaser 3 + Tauri 2 + Zustand 경계에 팀 합의
- char_a, char_b, 핵심 장소 5개 초안 확정

---

### M1. 공간/렌더링 수직 슬라이스

**목표**  
“마을 화면을 켜 두고 보고 싶다”는 감각의 최소 단위를 만든다.

**포함 범위**

- Phaser 기반 타일맵 로딩
- 카메라 이동/줌/선택 캐릭터 추적
- char_a 스프라이트 및 기본 이동 애니메이션
- 장소 3개 이상(집, 도서관, 카페)
- 충돌/진입점/상호작용 존
- 기본 월드 루프 연결

**산출물**

- 단일 연속 맵
- 캐릭터 A가 하루 루틴 일부를 따라 이동
- 시점/줌/장면 전환 감각 검증
- 맵/에셋 폴더 구조 확정

**종료 기준**

- A가 목적지를 따라 정상 이동한다
- 타일맵/카메라 구조가 이후 확장에 견딘다
- 마을 화면만으로도 최소한의 공간감이 느껴진다

---

### M2. 자율 시뮬레이션 코어

**목표**  
캐릭터가 “왜 움직이는지” 설명 가능한 구조를 만든다.

**포함 범위**

- WorldClock
- NeedSystem
- RoutinePlanner
- IntentScorer
- NavigationSystem
- ProjectionBuilder
- 헤드리스 시뮬레이션 테스트 하네스

**산출물**

- A의 하루 일정 생성
- 욕구 감소/회복
- 장소 선택 이유 태그 출력
- 1일 자동 시뮬레이션 가능

**종료 기준**

- A가 정지 루프 없이 하루를 보낼 수 있다
- 행동 선택 이유를 로그로 확인할 수 있다
- 헤드리스 1일 재생 테스트 통과

---

### M3. 사용자 입력 반영

**목표**  
“내 입력이 마을을 바꾼다”를 첫 번째로 체감시키는 단계.

**포함 범위**

- ActivityInputPanel
- PlayerActionEntry 저장
- InfluenceInterpreter
- 입력 후 일정 재평가
- 관련 아이콘/대사/장소 점유 변화 연출

**산출물**

- 공부/게임/운동/휴식 입력 4종
- 입력 후 A의 행동 변화
- 짧은 UI 피드백과 월드 피드백 연결

**종료 기준**

- 입력 후 3분 이내 가시적 변화가 1회 이상 나타난다
- 같은 입력을 여러 번 넣으면 누적 효과가 보인다
- 입력 없이도 게임은 계속 동작하지만, 입력 시 더 뚜렷한 변화가 생긴다

---

### M4. 관계/이벤트/해금

**목표**  
“살아 있는 마을”의 핵심인 상호작용 밀도를 만든다.

**포함 범위**

- char_b 해금
- RelationshipSystem
- EventDirector
- 기본 사회 이벤트 4~6종
- 카페/도서관/오락실/공원 상호작용
- 해금 알림/이벤트 포커스

**산출물**

- A와 B가 서로 만나고 관계가 변함
- 장면 기반 관계 변화
- 첫 번째 해금 루프 완성

**종료 기준**

- 1세션 안에 관계 이벤트가 최소 2회 이상 자연 발생
- char_b 해금이 설명 가능한 조건으로 일어난다
- 관계 변화가 행동 차이로 드러난다

---

### M5. 저장/안정화/콘텐츠 확장

**목표**  
실제 제품 골격으로 넘어갈 수 있는 기반을 만든다.

**포함 범위**

- 저장 슬롯 시스템
- snapshot + journal 저장
- 로드/복구
- char_c 해금
- 확장 시설 1~2종
- 장기 소크 테스트
- 디버그 오버레이

**산출물**

- 최소 2개 이상의 저장 슬롯
- 7일/30일 장기 테스트 결과
- 시설 확장 루프
- 장소 밀도 증가

**종료 기준**

- 저장/불러오기 후 상태 일치
- 비정상 종료 후 마지막 정상 스냅샷 복구 가능
- 7일 자동 시뮬레이션에서 치명 오류 없음

---

### M6. AI 보조 연출

**목표**  
LLM 없이도 동작하는 구조를 유지한 채, 표현의 질을 높인다.

**포함 범위**

- AIProvider 인터페이스
- MockProvider / Template fallback
- 짧은 대사
- 하루 일기
- 이벤트 요약
- 비용/지연/오프라인 fallback 정책

**산출물**

- 템플릿 기반 기본 일기
- AI 사용 시 더 풍부한 표현
- 캐시와 실패 복구

**종료 기준**

- AI 미연결 시도 제품이 완전 동작
- AI 연결 시 결과 품질이 체감 개선
- AI 실패가 게임 진행을 막지 않음

---

### M7. 후속 확장

**목표**  
제품형 확장을 검토한다.

**후보 항목**

- 클라우드 동기화
- 친구 마을 방문
- 비동기 공유
- 장식/꾸미기 볼륨 확장
- 유료 콘텐츠 패키징
- 실시간 멀티플레이 프로토타입

**주의점**

이 단계는 앞선 마일스톤의 핵심 재미가 검증된 후에만 진행한다.

## 5. 우선순위 게이트

다음 게이트를 넘지 못하면 후속 단계로 가지 않는다.

### Gate A: 공간감 검증

- 맵이 작더라도 “마을을 보고 있다”는 느낌이 드는가?
- 카메라가 자연스럽게 장면을 보여 주는가?

### Gate B: 자율성 검증

- A의 행동이 랜덤 순찰처럼 보이지 않는가?
- 하루 루프가 설명 가능한가?

### Gate C: 입력 인과 검증

- 공부/게임/운동/휴식 입력이 장면 차이를 만드는가?
- 입력 없이도 심심하지 않지만, 입력 시 더 재미있어지는가?

### Gate D: 관계 밀도 검증

- 관계가 숫자 표가 아니라 행동/장면으로 체감되는가?
- 캐릭터 B 해금이 설득력 있는가?

### Gate E: 제품 구조 검증

- 저장이 안정적인가?
- 장기 세션에서 시뮬레이션이 무너지지 않는가?

## 6. 트랙별 병행 작업 제안

### 6.1 엔지니어링 트랙

- sim-core
- renderer
- save system
- debug tools

### 6.2 콘텐츠/디자인 트랙

- 캐릭터 아키타입
- 이벤트 템플릿
- 해금 규칙
- 맵 배치와 동선

### 6.3 아트 트랙

- 타일셋
- 캐릭터 스프라이트
- 장소 소품
- 아이콘/오버레이

콘텐츠와 아트는 M1부터 함께 병행되어야 하며, 시스템이 다 된 뒤 한꺼번에 얹는 방식은 피한다.

## 7. 뒤로 미룰 기능

다음은 의도적으로 후순위로 둔다.

- 자동 앱 감지
- 복잡한 건설 UI
- 경제/재화 시스템
- 실시간 멀티플레이
- 대규모 캐릭터 수
- 오픈엔디드 자유 대화형 NPC
- 대형 대시보드/분석 화면

## 8. 마일스톤별 데모 체크포인트

| 마일스톤 | 외부 시연 가능 포인트 |
| --- | --- |
| M1 | 캐릭터 A가 마을을 걷는 장면 |
| M2 | 하루 루틴과 욕구 변화 |
| M3 | 사용자 입력 → 행동 변화 |
| M4 | A와 B의 사회 이벤트/해금 |
| M5 | 저장/확장 시설/장기 세션 |
| M6 | 일기/요약/연출 강화 |

## 9. 출시 준비 전 반드시 확인할 것

- 마을 화면이 계속 메인 화면인가?
- 정보 패널이 게임의 주인공이 되어 버리지 않았는가?
- 사용자의 입력이 월드에 실제 흔적을 남기는가?
- 가만히 두고 봐도 재밌는가?
- 새로운 캐릭터/시설 추가가 개발 지옥을 만들지 않는가?

## 10. 로드맵 승인 기준

- 기술/디자인/아트 팀이 단계 순서에 동의한다.
- M4 이전에 멀티플레이를 붙이지 않는다.
- M3 이전에 LLM 핵심 로직화를 시도하지 않는다.
- M5 이전에 수익화 기능을 넣지 않는다.
- M1~M4에서 “관찰 재미”를 최우선 지표로 삼는다.
