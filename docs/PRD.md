# PRD

문서 상태: Draft v0.1  
문서 목적: 제품 방향, 범위, 성공 기준 합의  
적용 범위: 데스크톱 우선, 싱글 플레이 우선, 시뮬레이션 중심

## 1. 제품 한 줄 정의

사용자의 생활 입력이 마을 주민들의 하루, 관계, 장소 사용 패턴에 영향을 주는 **공간 중심의 지속형 타운 시뮬레이션 게임**이다.  
핵심 재미는 “관리 화면을 보는 것”이 아니라, **픽셀 캐릭터들이 살아 있는 것처럼 마을을 돌아다니는 장면을 가만히 지켜보는 재미**에 있다.

## 2. 문제 정의

기존의 생산성/라이프로그 앱은 기록은 남기지만 감정적 보상이 약하다.  
반대로 많은 펫/아이들 게임은 귀엽지만 사용자의 실제 생활과 느슨하게만 연결된다.  

이 프로젝트는 다음 간극을 메운다.

1. 사용자의 실제 행동을 완전 자동 추적하지 않아도, 수동 입력만으로 충분히 “내 생활이 세계에 흔적을 남긴다”는 감각을 제공한다.
2. 보고만 있어도 재미있는 자율 시뮬레이션을 전면에 두고, 기록 행위는 그 시뮬레이션을 흔드는 입력으로 취급한다.
3. 시작은 매우 작게 하되, 장기적으로 캐릭터/장소/시설/꾸미기/유료 콘텐츠/멀티플레이까지 확장할 수 있는 구조를 만든다.

## 3. 제품 비전

### 3.1 비전 문장

“사용자의 하루가 작은 마을의 하루에 번져 나가는, 관찰형 생활 시뮬레이션.”

### 3.2 제품이 반드시 유지해야 하는 축

- **시뮬레이션 우선**: UI/대시보드가 아니라 마을과 캐릭터가 중심이다.
- **관찰 재미 우선**: 사용자가 적극 조작하지 않아도 볼거리가 계속 발생해야 한다.
- **공간감 우선**: 타일맵 공간이 항상 주인공이며, 프로필/차트 화면이 주 화면이 되어서는 안 된다.
- **점진적 인과성**: 입력 즉시 반응도 보여 주되, 일/주 단위의 누적 변화도 만들어야 한다.
- **장기 확장성**: 작은 프로토타입에서 끝나지 않도록, 콘텐츠와 기술 구조를 초기에 분리한다.

## 5. 타깃 사용자

### 5.1 1차 타깃

- 하루를 기록하고 싶지만, 숫자 앱보다 감정적인 피드백을 원한다.
- 화면을 켜 두고 구경하는 재미를 좋아한다.
- 동물/마을/관찰형 시뮬레이션, 아바타 공간, 코지 게임에 친화적이다.
- 복잡한 조작보다 가볍고 지속적인 관여를 선호한다.

### 5.2 2차 타깃

- 생산성 앱을 오래 못 쓰는 사용자
- 데스크톱에서 세컨드 스크린 성격의 게임을 켜 두는 사용자
- 자신만의 캐릭터/공간을 천천히 키우는 감각을 선호하는 사용자

## 6. 핵심 경험 원칙

### 6.1 플레이어가 느껴야 하는 감정

1. “내가 오늘 공부한 것이 진짜로 이 마을의 흐름을 바꿨다.”
2. “가만히 보고 있는데도 누가 어디 가고, 누구를 만나고, 무슨 일이 생겨서 재밌다.”
3. “정보창을 열기보다 마을 화면을 계속 켜 두고 싶다.”
4. “조금씩 마을이 넓어지고, 새로운 캐릭터가 들어오면서 세상이 커진다.”

### 6.2 반드시 보장해야 하는 경험 규칙

- 한 세션 초반 5분 안에 **의미 있는 장면 3개 이상**이 화면에 보여야 한다.
- 캐릭터는 단순 순찰이 아니라 **이유가 있는 이동**을 해야 한다.
- 사용자의 입력 후 **짧은 시간 내 체감 가능한 변화**가 최소 1개 이상 나타나야 한다.
- 관계 변화는 숫자만 오르내리는 것이 아니라 **장면, 대사, 회피/접근 행동**으로 드러나야 한다.
- 마을 화면에서 벗어나 정보를 확인하는 시간은 전체 세션의 30%를 넘지 않도록 설계한다.

## 7. 대표 사용자 시나리오

### 시나리오 A: 가벼운 저녁 세션

1. 사용자가 게임을 켠다.
2. 캐릭터 A가 집에서 나와 도서관 쪽으로 걷고, 카페 앞에서 잠시 멈춰 다른 캐릭터와 짧은 인사를 나눈다.
3. 사용자가 “공부 60분”을 입력한다.
4. 몇 분 안에 A의 오후 일정이 카페보다 도서관 쪽으로 기울고, 관련 감정 버프/대사/행동 변화가 발생한다.
5. 저녁에는 공원에서 짧은 휴식 이벤트가 생기고, 하루가 끝나며 일기나 요약 텍스트가 생성된다.
6. 다음 세션에는 누적된 입력으로 해금 조건이 조금씩 진행된다.

### 시나리오 B: 지속 플레이와 해금

1. 사용자가 며칠간 꾸준히 “게임”과 “휴식”을 입력한다.
2. 오락실 이용 빈도와 A의 놀이 욕구가 높아진다.
3. 특정 조건이 만족되면 캐릭터 B가 등장한다.
4. B는 오락실/카페 중심의 생활 패턴을 보이며, A와 상호작용을 시작한다.
5. 관계 이벤트와 장소 점유 패턴이 달라지며, 마을의 밀도가 증가한다.

## 9. MVP 정의

이 프로젝트는 단계적으로 개발한다. 여기서 말하는 **MVP**는 “장기 제품 구조를 버리지 않으면서도 가장 작은 재미 단위를 검증하는 버전”이다.

### 9.1 MVP 범위 (필수)

- 데스크톱 실행
- 단일 연속 마을 맵 1개
- 캐릭터 A 완전 동작
- 캐릭터 B 해금 1단계
- 장소 5개 이상
- 사용자 수동 입력 4종 이상 (공부, 게임, 운동, 휴식)
- 고정 틱 기반 자율 시뮬레이션
- 관계 수치 2축 이상 + 가시적 장면 반영
- 저장/불러오기
- 최소 1종의 일기/이벤트 요약 생성 (템플릿 기반 또는 LLM 보조)

### 9.2 MVP에서 제외

- 실시간 멀티플레이
- 자동 앱 감지
- 복잡한 건설/배치 UI
- 상점/재화/광범위한 경제 시스템
- 자유 채팅형 LLM NPC 대화
- 10명 이상 대규모 캐릭터 시뮬레이션

## 12. 성공 지표

### 12.1 프로덕트 감각 지표

- 세션 시작 후 2분 안에 캐릭터가 최소 1회 이상 의도 있는 이동을 한다.
- 세션 10분 기준, 가시적 상호작용 장면이 최소 4회 이상 발생한다.
- 사용자 입력 후 3분 안에 관련 장소/행동/아이콘/대사 중 하나 이상으로 결과가 보인다.
- 3~5세션 내에 신규 캐릭터 또는 시설 해금이 1회 이상 발생한다.

### 12.2 개발 단계 검증 지표

- 7일치 헤드리스 시뮬레이션에서 치명적 오류 없음
- 저장/불러오기 후 상태 불일치 없음
- 초기 맵에서 캐릭터 경로 막힘(stuck) 비율 1% 미만
- 관계 이벤트 템플릿이 특정 조합에 편중되지 않음

## 13. 출시 단위 정의

| 단계 | 목적 | 포함 범위 |
| --- | --- | --- |
| Vertical Slice | 핵심 감각 검증 | A 캐릭터, 1개 맵, 3개 장소, 입력 4종, 저장 |
| Alpha | 시스템 검증 | A/B, 관계 이벤트, 해금 1종, 확장 부지 연결 |
| Beta | 콘텐츠 검증 | A/B/C, 시설 확장 2종, 일기/요약, 장면 다양화 |
| 1.0 | 제품 골격 완성 | 안정성, 밸런스, 저장 마이그레이션, 장기 운영 기반 |

## 14. 운영/수익화 장기 방향 (현재 구현 범위 아님)

- 유료 캐릭터 스킨
- 유료 장소 테마팩
- 확장 지구/DLC
- 시즌성 장식 팩
- 친구 마을 방문형 비동기 기능
- 이후 실시간 멀티플레이 검토

현재 단계에서는 **수익화 기능을 게임 루프보다 먼저 만들지 않는다.**

## 15. 오픈 이슈

1. 하루 길이를 20분 내외로 둘지 30분 내외로 둘지 실제 플레이 테스트가 필요하다.
2. 사용자의 입력 카테고리 수를 최소 몇 개로 시작할지 확정이 필요하다.
3. char_b 해금 조건을 “입력 중심”으로 둘지 “관계 중심”으로 둘지 밸런싱이 필요하다.
4. 대사/일기 생성의 톤을 완전 템플릿으로 시작할지, 초기에 제한적 LLM을 붙일지 판단이 필요하다.
5. 마을을 완전 단일 맵으로 유지할지, 지구 단위 스트리밍으로 확장할지 후속 검토가 필요하다.

## 16. 승인 기준

이 문서는 아래가 합의되면 개발 착수 기준으로 본다.

- 핵심 재미가 “시뮬레이션 관찰”임에 팀이 동의한다.
- MVP에서 만들지 않을 것들이 명확하다.
- UI보다 공간이 우선이라는 원칙에 동의한다.
- 싱글 플레이 완성 전까지 멀티플레이를 미루는 것에 동의한다.
- LLM은 연출 보조이지 핵심 의사결정 엔진이 아니라는 데 동의한다.
