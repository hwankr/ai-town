# RISKS

문서 상태: Draft v0.1  
문서 목적: 장기 확장 가능한 시뮬레이션 중심 게임 개발에서 핵심 리스크와 대응책 정리

## 1. 리스크 관리 원칙

- 이 프로젝트의 가장 큰 리스크는 기술보다 **재미의 밀도**다.
- “될 것 같은 시스템”보다 “실제로 보이는 장면”을 우선 검증한다.
- 기능 추가보다 리스크 제거 순서로 개발한다.
- 리스크는 숨기지 말고, 조기 신호를 운영 지표처럼 본다.

## 2. 핵심 리스크 목록

| 우선순위 | 리스크 | 확률 | 영향도 | 요약 |
| --- | --- | --- | --- | --- |
| R1 | 자율 시뮬레이션이 실제로 재미없을 수 있음 | 높음 | 매우 큼 | 가장 중요한 제품 리스크 |
| R2 | 사용자 입력의 반영이 약해 체감되지 않을 수 있음 | 높음 | 큼 | 제품 차별점이 사라짐 |
| R3 | UI가 점점 대시보드화될 수 있음 | 중간 | 큼 | 방향성 붕괴 |
| R4 | 시스템이 과설계되어 MVP가 늦어질 수 있음 | 높음 | 큼 | 출시 지연 |
| R5 | 콘텐츠 제작량이 시스템 속도를 따라가지 못할 수 있음 | 높음 | 큼 | 볼거리 부족 |
| R6 | 경로 탐색/점유 처리로 캐릭터가 자주 막힐 수 있음 | 중간 | 큼 | 관찰 몰입 저해 |
| R7 | 저장 포맷/마이그레이션이 초기에 약하면 장기 운영이 힘들어짐 | 중간 | 큼 | 실제 제품 전환 저해 |
| R8 | LLM 연동이 품질/비용/지연 문제를 만들 수 있음 | 중간 | 중간 | 표현 품질 변동 |
| R9 | 자동 감지/멀티플레이 등 주변 기능이 코어를 잠식할 수 있음 | 높음 | 큼 | 스코프 붕괴 |
| R10 | 프라이버시 오해가 발생할 수 있음 | 중간 | 큼 | 신뢰 이슈 |

## 3. 상세 리스크 분석

### R1. 자율 시뮬레이션이 실제로 재미없을 수 있음

**설명**  
캐릭터가 움직이더라도 의미 없는 왕복이나 기계적 반복으로 보이면, 프로젝트의 핵심 가치가 성립하지 않는다.

**조기 신호**

- 플레이 테스트 중 사용자가 5분 내 다른 창으로 시선을 돌린다.
- 캐릭터의 이동 이유를 설명하기 어렵다.
- 이벤트는 있지만 기억에 남는 장면이 없다.
- 같은 장소 반복과 무의미한 대기가 많다.

**대응책**

- M1/M2부터 “장면 밀도”를 지표로 측정한다.
- EventDirector를 초기에 도입해 공백을 줄인다.
- 행동 후보 선정 로그를 확인해 이해 가능한 선택만 남긴다.
- 같은 행동 반복 시 점수 감쇠, 장소 다양성 보정 적용
- 한 세션 안에 작은 상호작용을 자주 배치한다.

**오너**  
Game Technical Director / System Designer

---

### R2. 사용자 입력의 반영이 약해 체감되지 않을 수 있음

**설명**  
사용자가 공부/게임/운동/휴식을 입력했는데 결과가 미미하면, 이 프로젝트의 고유성이 사라진다.

**조기 신호**

- 입력 후 3분이 지나도 눈에 띄는 변화가 없다.
- 사용자 피드백이 “그냥 원래 하던 행동 같다”로 나온다.
- 입력 결과가 숫자 UI에만 나타나고 월드에 안 보인다.

**대응책**

- 각 입력은 즉시성 효과 + 누적 효과를 동시에 갖게 한다.
- 장소 선호 변화, 이동 경로 변화, 감정 아이콘, 짧은 대사 등 월드 반영 우선
- A 캐릭터는 초반에 사용자 입력 영향이 특히 강하게 보이도록 설계
- 입력 결과에 대한 이유 태그를 디버그로 확인 가능하게 한다.

**오너**  
System Designer

---

### R3. UI가 점점 대시보드화될 수 있음

**설명**  
문제를 설명하려다 보면 각종 상태 패널이 늘어나고, 결국 타일맵보다 정보 패널을 더 오래 보게 될 위험이 있다.

**조기 신호**

- 스크린샷에서 마을보다 패널이 더 크다.
- 밸런스 조정 과정에서 모든 해답이 UI 추가로 귀결된다.
- 플레이어가 월드를 안 보고도 대부분 상태를 파악할 수 있다.

**대응책**

- “월드에서 먼저 보여 준다” 원칙을 디자인 리뷰 항목에 포함
- 패널은 보조 역할만 수행
- 말풍선/아이콘/장소 점유/행동 차이를 적극 사용
- 모든 신규 UI는 “이 정보를 월드에서 표현할 수 없는가?”를 먼저 검토

**오너**  
Game Technical Director / UX Owner

---

### R4. 시스템이 과설계되어 MVP가 늦어질 수 있음

**설명**  
장기 확장을 고려하다 보면, 실제 재미 검증 전에 추상화와 미래 대비 코드가 과도해질 수 있다.

**조기 신호**

- 아직 A 캐릭터도 안 움직이는데 인터페이스/추상화가 과하게 늘어남
- 네트워크/클라우드 관련 구조가 시뮬레이션 코어보다 먼저 커짐
- 툴링만 많고 플레이 가능한 빌드가 늦다

**대응책**

- 모든 추상화는 M1~M4의 실제 사용처가 있을 때만 추가
- `InProcessSimulationHost`로 먼저 만들고, worker는 후속 단계에서 연결
- 멀티플레이와 클라우드 동기화는 인터페이스만 남기고 구현 보류
- 마일스톤 게이트를 넘지 못하면 후속 확장 금지

**오너**  
Staff Engineer

---

### R5. 콘텐츠 제작량이 시스템 속도를 따라가지 못할 수 있음

**설명**  
시스템이 있어도 장소, 이벤트, 대사, 캐릭터 개성이 부족하면 관찰 재미가 금방 소진된다.

**조기 신호**

- 같은 이벤트가 반복되어 보임
- 장소가 시각적으로 구분되지 않음
- 캐릭터 간 차이가 행동보다 설명문에서만 느껴짐

**대응책**

- 이벤트 템플릿을 데이터 드리븐으로 설계
- “작은 장면” 중심 콘텐츠를 많이 만든다
- 장소/행동 태그 재조합으로 이벤트 재사용성 확보
- char_a/char_b/char_c의 차이를 아트와 행동 둘 다에서 드러낸다

**오너**  
Content Designer / Art Lead

---

### R6. 경로 탐색/점유 처리 문제

**설명**  
타일맵 공간에서 여러 캐릭터가 이동하면 막힘, 왕복 떨림, 겹침 문제가 쉽게 발생한다.

**조기 신호**

- 문 앞에서 캐릭터가 흔들리거나 멈춤
- 특정 장소 입구에 병목이 심함
- 좌석/상호작용 포인트 경쟁이 자주 꼬임

**대응책**

- 맵 설계 단계에서 진입점과 폭 확보
- 단순 타일 A*만 믿지 않고, 장소 앵커/예약 시스템 도입
- 자주 쓰는 경로는 캐시 또는 고수준 zone graph 사용
- stuck 감지 후 재계산/대체 행동 fallback 제공

**오너**  
Staff Engineer / Tech Director

---

### R7. 저장 포맷/마이그레이션 리스크

**설명**  
초기에 저장 구조를 가볍게 만들면, 나중에 콘텐츠 추가 시 호환성 문제가 크게 터질 수 있다.

**조기 신호**

- 저장 파일에 버전이 없다.
- UI 상태와 월드 상태가 섞여 있다.
- 새 필드 추가 때 이전 저장이 자주 깨진다.

**대응책**

- snapshot + journal 구조 유지
- saveVersion/contentVersion 분리
- SaveMigration 모듈을 초기부터 둔다
- 자동 마이그레이션 테스트 작성

**오너**  
Staff Engineer

---

### R8. LLM 연동 품질/비용/지연 문제

**설명**  
대사/일기 품질이 들쭉날쭉하거나 응답이 늦으면 오히려 제품 몰입을 해칠 수 있다.

**조기 신호**

- 같은 상황에서 톤이 흔들림
- 텍스트가 캐릭터 개성을 무시함
- 비용 때문에 호출량을 못 유지함

**대응책**

- 템플릿 기반 fallback을 기본으로 준비
- AI는 비동기 결과 또는 사후 요약 중심으로 사용
- 짧은 프롬프트, 강한 톤 가이드, 캐싱 사용
- 핵심 게임 루프를 AI 응답 대기 상태로 만들지 않음

**오너**  
AI/Tools Owner

---

### R9. 주변 기능이 코어를 잠식할 수 있음

**설명**  
자동 감지, 멀티플레이, 꾸미기, 수익화 같은 주변 요소가 빨리 커지면 핵심 시뮬레이션이 약해진다.

**조기 신호**

- 백로그 상위 항목에 코어 외 기능이 많아짐
- 플레이 테스트 피드백이 “재밌다”보다 “기능이 많다/없다”로 흘러감
- 핵심 루프 완성 전 상점/결제/친구 기능 논의가 많아짐

**대응책**

- 코어 게이트 통과 전 후순위 기능 개발 금지
- 로드맵에서 명시적으로 M5 이후로 미룸
- 모든 신규 기능 제안에 “시뮬레이션 중심성 유지 여부” 평가 항목 추가

**오너**  
Product Owner / Tech Director

---

### R10. 프라이버시 오해

**설명**  
사용자는 실제 생활 입력을 넣기 때문에, 자동 추적을 하지 않아도 “뭔가 감시당하나?”라고 느낄 수 있다.

**조기 신호**

- 사용자 문의가 데이터 수집 방식에 집중됨
- 저장 위치/동기화 여부를 혼동함
- 자동 감지 기능이 있다고 오해함

**대응책**

- “수동 입력, 로컬 저장, 자동 추적 없음”을 명확히 표시
- 데이터 경계 문구를 온보딩과 설정에 노출
- 클라우드 동기화 도입 전까지 외부 전송 없음 원칙 유지

**오너**  
Product Owner

## 4. 리스크 우선 대응 순서

1. R1 자율 시뮬레이션 재미
2. R2 입력 체감
3. R3 공간 중심 UX 유지
4. R4 과설계 방지
5. R6 경로/점유 안정성
6. R5 콘텐츠 생산성
7. R7 저장 안정성
8. R8 AI 표현 품질
9. R9 주변 기능 팽창
10. R10 프라이버시 메시징

## 5. 리스크 리뷰 주기

| 주기 | 리뷰 항목 |
| --- | --- |
| 주간 | 장면 밀도, 입력 체감, 막힘/stuck 사례 |
| 마일스톤 종료 시 | 구조 과설계 여부, 저장 안정성, 이벤트 반복률 |
| 주요 플레이 테스트 후 | 재미 저하 지점, UI 과다 여부, 이해 가능성 |

## 6. 레드라인

아래가 발생하면 일정과 별개로 즉시 수정 우선순위를 높인다.

- 5분 관찰 내 의미 있는 장면이 거의 없음
- 입력 후 체감 변화가 없음
- 저장 손상으로 플레이 진행이 막힘
- 캐릭터 stuck가 반복적으로 발생
- 마을 화면보다 패널 화면이 더 오래 열려 있음

## 7. 리스크 관리 성공 기준

- 핵심 리스크(R1~R4)에 대해 플레이 테스트에서 개선이 보인다.
- 기술 리스크가 재미 리스크를 가리지 않는다.
- 기능을 더하는 대신 방향을 더 선명하게 만든다.
