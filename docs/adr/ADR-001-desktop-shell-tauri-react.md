# ADR-001: 데스크톱 셸로 Tauri 2 + React + TypeScript를 채택한다

상태: Proposed  
작성일: 2026-03-08  
결정자: Staff Engineer / Game Technical Director / System Designer

## 1. 맥락

이 프로젝트는 다음 조건을 동시에 만족해야 한다.

- 데스크톱 우선
- 로컬 우선 저장
- 장기적으로 실제 제품 수준 배포 가능
- 공간 중심의 타일맵 게임 화면과 일반 앱 UI가 공존
- 네이티브 권한은 최소화하면서 파일 시스템 접근 필요
- 인디 팀이 유지 가능한 웹 기술 기반 워크플로가 필요

Electron도 가능하지만 패키지 크기, 메모리 풋프린트, 배포 감각, 네이티브 권한 분리 방식 측면에서 더 가벼운 선택이 필요하다.  
Godot는 게임 엔진으로 매우 강력하지만, 현재 방향인 React/TypeScript 중심 생산성과 데스크톱 앱 셸 요구와는 거리가 있다.

## 2. 결정

앱 셸은 **Tauri 2**를 사용하고, 프런트엔드는 **React + TypeScript**로 구현한다.

- Tauri 2는 앱 패키징, 파일 시스템 접근, 향후 업데이터와 네이티브 권한 경계를 담당한다.
- React는 설정, 저장 슬롯, 활동 입력, 알림 등 앱형 UI를 담당한다.
- TypeScript는 프런트엔드와 시뮬레이션 패키지 전반의 공통 언어로 사용한다.
- Rust는 Tauri 커맨드와 네이티브 경계에만 최소한으로 사용하고, 도메인 로직은 TypeScript 쪽에 둔다.

## 3. 근거

### 3.1 왜 이 선택이 프로젝트에 맞는가

1. **데스크톱 우선 + 로컬 우선** 요구와 잘 맞는다.  
   저장 파일, 설정 파일, 향후 업데이터를 네이티브 앱처럼 다루기 쉽다.

2. **월드 화면 + 앱 UI 조합**에 유리하다.  
   React는 입력/설정/슬롯 UI를 담당하고, 월드 렌더링은 Phaser로 분리할 수 있다.

3. **소규모 팀의 생산성**이 높다.  
   TypeScript 중심 개발로 sim-core, UI, 브리지 계층의 언어를 통일할 수 있다.

4. **권한 경계가 분명하다.**  
   파일 접근과 네이티브 기능을 Tauri capability와 command 계층으로 국소화할 수 있다.

## 4. 적용 범위

이 결정은 아래에 적용된다.

- `apps/desktop`
- `apps/desktop/src-tauri`
- `apps/desktop/src/app`
- `apps/desktop/src/bridges`

이 결정은 아래에 직접 적용되지 않는다.

- `packages/sim-core`의 도메인 로직 구현 방식
- 렌더러 선택(별도 ADR)
- AI 연동 정책(별도 ADR)

## 5. 결과

### 5.1 기대 효과

- 가벼운 데스크톱 패키징
- 로컬 파일 저장 구조 확립
- React 기반 UI 생산성 확보
- 권위 시뮬레이션과 앱 셸의 역할 분리

### 5.2 비용/단점

- Tauri와 Rust command 계층을 별도로 관리해야 한다.
- 웹뷰 환경 차이로 인한 플랫폼별 렌더링 검증이 필요하다.
- 프런트엔드와 네이티브 경계에서 DTO 설계가 요구된다.

## 6. 대안 검토

### 대안 A: Electron + React

**장점**

- 웹 개발자에게 익숙함
- 풍부한 레퍼런스

**보류 이유**

- 앱 크기와 런타임 풋프린트가 상대적으로 크다.
- 현재 프로젝트는 브라우저 전체를 새로 번들링하는 무게보다, 타일맵 시뮬레이션의 응답성과 장기 배포 감각이 중요하다.

### 대안 B: Godot 단일 엔진

**장점**

- 게임 제작 툴체인 통합
- 2D 기능 강력

**보류 이유**

- React/TypeScript 중심 앱 UI와 레포 구조를 유지하기 어렵다.
- 팀이 목표로 하는 데스크톱 앱형 제품 구조와 기술 방향에서 벗어난다.

### 대안 C: 웹 앱(PWA) 우선

**장점**

- 배포가 쉽다.
- 플랫폼 의존성이 낮다.

**보류 이유**

- 파일 시스템 제어, 로컬 저장 구조, 업데이트 정책, 향후 네이티브 기능 확장에 제약이 생긴다.
- 현재 목표는 웹보다 데스크톱 우선이다.

## 7. 후속 작업

- Tauri command 경계 최소화
- 저장소/설정 저장은 src-tauri 쪽으로 집중
- capability 정책을 저장 경로 중심으로 좁게 설계
- sim-core에는 Tauri 의존성을 절대 넣지 않기

## 8. 재검토 조건

아래 상황이 오면 이 결정을 재검토한다.

- 데스크톱보다 모바일이 핵심 우선순위가 될 때
- 네이티브 성능 요구가 급증해 핵심 로직을 Rust로 이전해야 할 때
- React UI보다 게임 엔진 단일 UI가 압도적으로 유리해질 때
