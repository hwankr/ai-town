# ADR-002: 월드 렌더러로 Phaser 3를 채택한다

상태: Proposed  
작성일: 2026-03-08  
결정자: Staff Engineer / Game Technical Director / System Designer

## 1. 맥락

이 프로젝트의 메인 화면은 Gather.town 같은 **타일맵형 픽셀 공간**이어야 한다.  
다음이 필수다.

- 타일맵 로딩과 레이어 구조
- 카메라 이동, 추적, 줌, 이벤트 포커스
- 다수 캐릭터 스프라이트 애니메이션
- 씬 분리
- React/Tauri 앱 셸과의 공존
- 작은 팀이 장기 유지 가능한 개발 난도

렌더러는 “멋진 그래픽”보다 **관찰하기 좋은 공간과 캐릭터 상태 가시화**에 최적화되어야 한다.

## 2. 결정

월드 렌더러는 **Phaser 3**를 기본안으로 채택한다.

- `WorldScene`, `OverlayScene`, `EventScene` 중심 구조를 사용한다.
- Phaser는 타일맵, 카메라, 스프라이트, 월드 내부 오버레이 표현에 집중한다.
- 도메인 의사결정은 sim-core가 수행하고, Phaser는 projection을 소비한다.
- Phaser 3를 우선 채택하고, Phaser 4는 추후 생태계 안정화 시 별도 검토한다.

## 3. 근거

### 3.1 타일맵

이 프로젝트는 장소 이동, 골목, 실내, 공원, 오락실 같은 공간이 핵심이다.  
Phaser 3는 타일맵과 타일 레이어를 다루는 기본기가 충분하고, Tiled 기반 워크플로와도 잘 맞는다.  
이 덕분에 아트/맵 제작 파이프라인을 복잡한 커스텀 엔진 작업 없이 시작할 수 있다.

### 3.2 카메라

이 게임의 재미는 “지금 재밌는 장면을 보여 주는 것”과 밀접하다.  
Phaser의 카메라 기능은 다음 목적에 적합하다.

- 선택 캐릭터 추적
- 중요 이벤트 포커스
- 부드러운 팬/줌
- 씬별 카메라 분리

카메라를 엔진 레벨에서 안정적으로 다룰 수 있다는 점이 중요하다.

### 3.3 애니메이션

캐릭터는 작지만 생동감이 있어야 한다.  
Phaser는 2D 스프라이트 애니메이션, 상태 전환, 다수 오브젝트 갱신을 관리하기 편하다.  
이 프로젝트가 필요로 하는 것은 복잡한 스켈레탈 애니메이션이 아니라, 명확하고 읽기 쉬운 2D 루프 애니메이션이다.

### 3.4 씬 관리

월드, 오버레이, 이벤트 연출을 분리해야 한다.  
Phaser의 씬 구조는 이 요구에 잘 맞으며, 앱 전체를 Phaser로 덮지 않고도 필요한 부분만 사용할 수 있다.

## 4. 대안 검토

### 대안 A: PixiJS

**장점**

- 렌더링 성능이 좋고 유연하다.

**보류 이유**

- 타일맵, 씬 라이프사이클, 카메라 규약, 게임 루프 보조 구조를 많이 직접 설계해야 한다.
- 인디 팀의 초기 구현 비용이 올라간다.

### 대안 B: React Canvas / DOM 기반 직접 구현

**장점**

- 앱 UI와 구조를 한 곳에 모을 수 있다.

**보류 이유**

- 타일맵/카메라/스프라이트/애니메이션이 핵심인 프로젝트에는 부적합하다.
- 결과적으로 “렌더러를 직접 만드는 비용”을 떠안게 된다.

### 대안 C: Godot

**장점**

- 게임 엔진으로 기능이 강력하다.

**보류 이유**

- 현재의 Tauri + React + TypeScript 중심 구조와 맞지 않는다.
- 제품형 데스크톱 앱 셸과 게임 엔진 UI를 혼합하는 복잡도가 커진다.

### 대안 D: Phaser 4

**장점**

- 장기적으로 흥미롭다.

**보류 이유**

- 현재 단계에서는 Phaser 3가 더 안정적인 선택이다.
- 프로덕션 기준으로는 지금 바로 쓸 수 있는 성숙한 버전을 우선한다.

## 5. 결과

### 5.1 기대 효과

- 타일맵 공간감을 빠르게 확보
- 카메라 연출을 구조적으로 관리
- 캐릭터 애니메이션 구현 단순화
- 월드/오버레이/이벤트 씬 분리 가능
- React/Tauri 앱 셸과 충돌이 적음

### 5.2 비용/단점

- Phaser가 도메인 로직을 잠식하지 않도록 경계 관리가 필요하다.
- React와 Phaser의 상태 동기화 전략을 명확히 해야 한다.
- 지나치게 Phaser 내부 객체에 의존하면 테스트가 어려워질 수 있다.

## 6. 적용 규칙

- Phaser 씬은 authoritative state를 직접 계산하지 않는다.
- 캐릭터의 위치와 행동은 projection을 통해 주입받는다.
- 월드 내부 UI(아이콘, 말풍선, 선택 마커)는 Phaser에서 처리한다.
- 패널형 UI(입력 폼, 설정, 슬롯)는 React에서 처리한다.
- 렌더러는 보간과 연출 상태만 가진다.

## 7. 후속 작업

- `packages/game-renderer` 패키지 생성
- `BootScene`, `WorldScene`, `OverlayScene`, `EventScene` 기본 뼈대 작성
- Tiled 맵 파이프라인 확정
- 카메라 포커스 정책 문서화
- projection schema 확정

## 8. 재검토 조건

- Phaser가 다수 캐릭터/맵 규모에서 병목이 심할 때
- React/Tauri와의 통합 비용이 지나치게 커질 때
- 프로젝트 방향이 고급 게임 엔진 중심으로 바뀔 때
