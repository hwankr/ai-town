# ADR-004: 초기 단계에서 LLM은 비권위적 표현 계층에만 사용한다

상태: Proposed  
작성일: 2026-03-08  
결정자: Staff Engineer / Game Technical Director / System Designer

## 1. 맥락

프로젝트의 핵심 매력은 “보고만 있어도 재미있는 자율 시뮬레이션”이다.  
LLM은 매력적인 대사와 일기를 만들 수 있지만, 초반부터 행동 결정 엔진으로 넣으면 다음 문제가 생긴다.

- 일관성 저하
- 비용 예측 어려움
- 지연 시간
- 디버깅 어려움
- 저장/재현 불가능성 증가
- 콘텐츠 톤 흔들림

따라서 LLM은 초기에 핵심 의사결정자가 아니라 **연출 보조자**로 쓰는 것이 합리적이다.

## 2. 결정

초기 단계(Vertical Slice ~ Beta)에서 LLM은 아래 용도로만 사용한다.

- 짧은 대사 생성 또는 재작성
- 하루 일기
- 이벤트 하이라이트 요약
- 해금/연출용 문안
- 관계 변화의 표현 보강

LLM은 아래 영역에는 사용하지 않는다.

- 행동 점수 계산
- 루틴 생성의 권위 결정
- 이동/경로 계산
- 해금 판정
- 저장 원본 상태 변경
- 게임 진행에 필수적인 응답

## 3. 근거

### 3.1 핵심 재미는 규칙 기반으로 먼저 검증해야 한다

이 프로젝트가 성공하려면, AI 문장이 좋아서가 아니라 캐릭터가 실제로 살아 보이기 때문이어야 한다.  
따라서 욕구, 루틴, 관계, 이벤트는 deterministic하고 테스트 가능한 규칙 기반으로 먼저 만들어야 한다.

### 3.2 실패 시 fallback이 가능해야 한다

LLM 응답이 없거나 느리거나 비싸더라도 게임은 정상 진행되어야 한다.  
표현 계층에만 사용하면 템플릿 기반 fallback으로 품질 저하만 감수하고 진행은 유지할 수 있다.

### 3.3 톤과 개성 관리가 쉬워진다

캐릭터별 말투/개성은 content-data와 tone policy로 관리하고, LLM은 그 틀을 보강하는 역할을 한다.  
이 방식이 초기 제품 품질 통제에 유리하다.

## 4. 결과

### 4.1 기대 효과

- 핵심 시뮬레이션의 디버깅 가능성 유지
- 비용/지연 리스크 축소
- AI 제거 시에도 제품 유지 가능
- 일기/대사/요약에서 감정적 보상 강화

### 4.2 비용/단점

- 초기에는 “진짜 AI NPC”처럼 느껴지는 자유도는 제한적이다.
- 표현 품질이 템플릿보다 조금 나아지는 수준에 머무를 수 있다.
- 강한 개인화 대화를 기대하는 일부 사용자에게는 약하게 느껴질 수 있다.

## 5. 대안 검토

### 대안 A: 행동 결정까지 LLM 사용

**장점**

- 겉보기에는 지능적이고 신선해 보일 수 있다.

**보류 이유**

- 일관성, 재현성, 비용, 디버깅 모두 악화된다.
- 초기 핵심 재미 검증을 방해한다.

### 대안 B: LLM을 전혀 사용하지 않음

**장점**

- 구조가 단순하고 비용이 없다.

**보류 이유**

- 장기적으로 일기/요약/대사 표현의 감정적 보상을 강화할 기회를 놓친다.
- 이 프로젝트의 “지속형 AI 타운” 정체성을 일부 약화시킨다.

### 대안 C: 전부 로컬 소형 모델

**장점**

- 프라이버시와 오프라인 측면의 장점 가능

**보류 이유**

- 초기 팀 규모와 품질 목표에 비해 복잡도가 높다.
- 플랫폼별 모델 배포/용량/성능 이슈가 크다.

## 6. 적용 규칙

- AI 결과는 항상 비권위 artifact로 저장하거나 캐시한다.
- AI가 생성한 텍스트는 sim-core 상태를 직접 수정하지 않는다.
- AI 입력 컨텍스트는 이벤트/관계/톤 태그의 요약만 사용한다.
- 실패 시 템플릿으로 즉시 대체한다.
- 비용이 민감한 구간에는 호출 빈도 제한을 둔다.

## 7. 후속 작업

- `AIProvider` 인터페이스 정의
- `TemplateDialogueComposer`, `TemplateDiaryComposer` 먼저 구현
- 캐릭터 tone tags와 event summary seed 정의
- artifact cache와 timeout 정책 추가
- 오프라인/실패 시 UX 메시지 정리

## 8. 재검토 조건

- 규칙 기반 시뮬레이션이 안정화되어 행동 다양성 한계가 명확해질 때
- 비용/지연 문제를 감당 가능한 범위에서 운영할 수 있을 때
- 특정 기능(예: 장면 대사 확장)에 한정해 더 깊은 AI 개입이 필요할 때
