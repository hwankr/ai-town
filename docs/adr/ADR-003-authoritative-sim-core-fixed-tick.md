# ADR-003: 권위 시뮬레이션 코어를 순수 로직 모듈로 분리하고 고정 틱으로 운영한다

상태: Proposed  
작성일: 2026-03-08  
결정자: Staff Engineer / Game Technical Director / System Designer

## 1. 맥락

이 프로젝트의 핵심은 “캐릭터가 실제로 생활한다”는 느낌이다.  
이 재미를 만들려면 다음이 필수다.

- 행동 선택 이유를 설명 가능해야 한다.
- 저장/불러오기 후에도 상태가 재현 가능해야 한다.
- 렌더 프레임과 상관없이 시뮬레이션이 안정적으로 돌아야 한다.
- 장기적으로 헤드리스 테스트, 리플레이, 멀티플레이 확장 가능성을 확보해야 한다.

만약 게임 로직을 Phaser 씬이나 React 상태 안에 섞어 넣으면, 디버깅/저장/재현/확장성이 급격히 악화된다.

## 2. 결정

권위 시뮬레이션은 `packages/sim-core`의 **순수 TypeScript 로직 모듈**로 분리한다.

- sim-core는 React, Phaser, Tauri 의존성이 없다.
- 시뮬레이션은 **고정 틱** 방식으로 진행한다.
- 입력은 Command Queue로 들어오고, 결과는 Event와 Projection으로 나온다.
- 렌더 프레임은 시뮬레이션과 별도로 보간만 수행한다.
- 초기 버전은 in-process로 시작하되, 구조적으로는 worker-host 전환이 가능하게 만든다.

## 3. 근거

### 3.1 재현성

고정 틱 + 시드 RNG 구조는 동일 입력 시 동일 결과를 재생하기 쉽다.  
이는 다음에 중요하다.

- 저장/복구
- 버그 재현
- 밸런스 테스트
- 장기 소크 테스트

### 3.2 설명 가능성

이 프로젝트는 캐릭터가 왜 특정 장소로 갔는지, 왜 누구를 피했는지 설명 가능한 것이 중요하다.  
순수 로직 모듈로 분리하면 행동 점수화, 관계 변화, 이벤트 발생 조건을 독립 테스트할 수 있다.

### 3.3 렌더링과의 분리

렌더링 프레임 속도는 장치/환경에 따라 다르다.  
시뮬레이션을 렌더 프레임에 묶으면 결과가 흔들리고 디버깅이 어려워진다.  
고정 틱 시뮬레이션과 렌더 보간 분리는 이 문제를 줄인다.

### 3.4 미래 확장성

훗날 멀티플레이 또는 서버형 시뮬레이션을 검토할 때도, 이미 command/state/event 경계가 있으면 이전 비용이 작다.

## 4. 결과

### 4.1 기대 효과

- 테스트 용이성 상승
- 저장/마이그레이션 구조 명확화
- 렌더러 교체 가능성 확보
- 장기 소크 테스트 가능
- AI/클라우드/멀티플레이와의 경계 확보

### 4.2 비용/단점

- Projection 계층을 별도로 설계해야 한다.
- React/Phaser와의 연결 코드가 늘어난다.
- 초기에는 “그냥 씬 안에서 처리하면 빨랐을 일”이 느리게 느껴질 수 있다.

## 5. 대안 검토

### 대안 A: Phaser 씬이 권위 상태를 직접 보유

**장점**

- 구현 속도가 빠를 수 있다.

**보류 이유**

- 저장, 테스트, 리플레이, 렌더링 독립성이 모두 악화된다.
- 장기 제품 구조에 맞지 않는다.

### 대안 B: React/Zustand가 게임 원본 상태를 직접 보유

**장점**

- UI와 상태 연결이 단순해 보인다.

**보류 이유**

- 게임 권위 상태와 UI 상태가 섞인다.
- 대규모 갱신과 성능/구조 문제가 쉽게 생긴다.
- 멀티플레이/리플레이/저장 구조가 약해진다.

### 대안 C: ECS를 처음부터 전면 도입

**장점**

- 성능과 데이터 지향 구조에 장점이 있다.

**보류 이유**

- 현재 규모와 팀 구성에서 복잡도가 과하다.
- 우선 필요한 것은 관찰 재미 검증이지, 아키텍처 실험이 아니다.

## 6. 운영 규칙

- sim-core에서 도메인 규칙을 결정한다.
- projection은 읽기 전용 뷰 모델이다.
- 렌더러는 projection만 보고 움직인다.
- UI 스토어는 권위 상태를 소유하지 않는다.
- 모든 외부 입력은 Command 형태로 정규화한다.

## 7. 고정 틱 정책

- 기본 1x 속도에서 현실 1초 = 게임 1분
- 내부는 초당 4틱
- 1틱 = 게임 15초
- 이동, 욕구, 감정, 이벤트 평가 모두 틱 기반으로 계산
- 렌더러는 틱 사이의 위치를 보간해 부드럽게 보이게 함

## 8. 후속 작업

- `SimRunner`, `WorldClock`, `CommandQueue` 기본 틀 작성
- Projection schema 정의
- 헤드리스 시뮬레이션 테스트 하네스 마련
- 저장 serializer와 replay harness 연결
- `InProcessSimulationHost`와 `WorkerSimulationHost` 인터페이스 동일화

## 9. 재검토 조건

- 캐릭터 수와 맵 규모가 커져 현재 구조가 명백히 비효율적일 때
- 서버 권위형 구조로 빠르게 전환해야 할 때
- sim-core의 TypeScript 구현이 유지보수상 한계에 도달했을 때
